# `/grill` 提示词模板

> 复制下方模板到 Cursor 对话，替换 `【...】` 占位符后发送。

---

## 模板 A — 通用需求盘问

```markdown
/grill

## 背景
【用 1-3 句话描述你要做什么，例如：在 RK3568 上新增 I2C 传感器采集模块】

## 当前想法（如有）
【你的初步设计、约束、或不确定的点】

## 指令
请对我进行 **Grill（需求盘问）**，严格遵守：

1. **一次只问一个问题**，等我回答后再继续
2. 每个问题给出你的**推荐答案**（基于代码库或 C++/嵌入式最佳实践）
3. 能从代码库找到答案的问题，**先读代码库**，不要问我
4. 沿设计决策树逐分支推进，先解决依赖决策
5. 关注：行为边界、公共接口（头文件）、错误处理、资源所有权、线程/ISR、硬件依赖、CMake 影响、测试策略

## 结束条件
当你认为需求已足够清晰时，输出 **Grill 共识摘要**，包含：
- 公共接口草案（头文件签名）
- 按优先级排列的行为清单（可转化为测试名）
- CMakeLists.txt 变更范围
- 未决事项（如有）

**在我确认摘要前，不要写任何实现代码。**
```

## 模板 B — 带文档产出的 Grill（grill-with-docs）

```markdown
/grill

## 背景
【任务描述】

## 指令
请进行 Grill 需求盘问（一次一问，每问给推荐答案）。

额外要求：
- 若项目有 `CONTEXT.md`，盘问时对照术语表，发现冲突立即指出
- 术语确认后，**即时更新** `CONTEXT.md`（只写领域词汇，不写实现细节）
- 遇到难逆转的架构取舍时，提议创建 ADR（`docs/adr/NNNN-标题.md`）

结束时输出 Grill 共识摘要，等我确认后再进入 TDD。
```

## 模板 C — 快速 Grill（小功能）

```markdown
/grill --quick

## 要做的改动
【一句话描述，例如：给 SensorReader 加重试逻辑】

## 限制
- 最多问 **5 个问题**
- 每个问题带推荐答案
- 结束后给出：接口变更、行为清单、CMake 变更
- 我确认后开始 TDD
```

## 模板 D — 嵌入式硬件相关 Grill

```markdown
/grill

## 背景
【硬件/驱动/中间件任务描述】

## 已知硬件信息
- 平台：【如 RK3568 / MSP0 / 裸机】
- 接口：【I2C / SPI / GPIO / Mailbox / UART】
- 相关文件：【@头文件或文档路径】

## 指令
Grill 需求盘问，额外必须澄清：
1. 运行环境（Linux 用户态 / 内核驱动 / 裸机 / FreeRTOS）
2. 错误处理策略（错误码 / 返回值 / 日志 + 降级）
3. 硬件访问方式（sysfs / devmem / ioctl / 直接寄存器）
4. 测试策略（Fake 硬件 / 主机单元测试 / 板上集成测试）
5. CMake target 结构（static lib / shared lib / executable）

一次一问，给推荐答案。结束后输出共识摘要。
```

---

## 使用建议

| 场景 | 推荐模板 |
|------|----------|
| 新模块 / 不确定的需求 | 模板 A |
| 长期项目，需维护术语 | 模板 B |
| 小改动 / 时间紧 | 模板 C |
| 驱动 / 嵌入式 / 硬件 | 模板 D |
